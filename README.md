# LOGO Language Server

A Language Server Protocol (LSP) implementation for the LOGO programming language, written in Kotlin. Built with LSP4IJ and ANTLR4.

## Features

- **Syntax highlighting** — semantic tokens for keywords (including turtle commands), procedures, variables, and numbers
- **Go-to-declaration** — jump to procedure and variable definitions
- **Diagnostics** — undefined variables, unknown procedures, and syntax errors
- **Rename refactoring** — procedures, variables, and parameters with correct scoping
- **Change signature refactoring** — parameter reordering with automatic update of all call sites

---

## Focus

This project focuses on implementing **refactoring support in LSP**, specifically:
- symbol resolution
- reference tracking
- multi-location program transformations (rename, change signature)

rather than syntax-level features.

## Prerequisites

- Java 17 or later
- No other dependencies required — the fat JAR bundles everything

## Running

```bash
java -jar build/libs/jb-logo-lsp-all.jar
```

The server communicates over **stdio** (stdin/stdout). It blocks waiting for LSP messages — do not run it directly in a terminal. Connect it through an LSP client as described below.

---

## Connecting to LSP4IJ (IntelliJ IDEA)

LSP4IJ is a generic LSP client plugin for IntelliJ IDEA.

**Step 1 — Install LSP4IJ**

Settings → Plugins → Marketplace → search **LSP4IJ** → Install → Restart IDE

**Step 2 — Register the server**

Settings → Languages & Frameworks → Language Servers → click **+**

| Field | Value |
|---|---|
| Name | Logo LSP |
| Command | `java -jar /absolute/path/to/jb-logo-lsp-all.jar` |

Use the full absolute path to the JAR.

**Step 3 — Map to .logo files**

In the same dialog, go to the **Mappings** tab → click **+**

| Field | Value |
|---|---|
| File name pattern | `*.logo` |

**Step 4 — Open a .logo file**

Create or open any file ending in `.logo` inside an IntelliJ project. The server starts automatically. You can monitor server activity in **View → Tool Windows → LSP Consoles**.

## Usage

Open the included `sample.logo` file to see all features in action.

**Syntax highlighting** appears automatically when a `.logo` file is opened.

**Go-to-declaration** — place the cursor on any `:variable` reference or procedure call and press Ctrl+click.

**Diagnostics** — red squiggles appear on undefined variables; yellow squiggles on unknown procedure calls. They update in real time as you type.

**Rename** — place the cursor on any procedure name, variable, or parameter and press Shift+F6. Parameter renames are scoped to the enclosing procedure; procedure and variable renames update all references across the file.

**Change signature** — place the cursor on a parameter name in a `TO` definition line, press Alt+Enter, and select **"Change Signature: Reverse Parameter Order"**. This reverses the parameter order and updates all call sites atomically.

## Architecture

```
Logo.g4  (ANTLR grammar)
   │
   ▼
LogoLexer + LogoParser  (generated by ANTLR)
   │
   ▼
DocumentStore  (parses on open/change, stores ParsedDocument per URI)
   │
   ├── SymbolTableBuilder  (walks parse tree, records procedure/variable definitions)
   │
   ├── SemanticTokensProvider  (walks parse tree, emits LSP semantic token data)
   │
   ├── DiagnosticEngine  (walks parse tree, checks references against symbol table)
   │
   └── LSP Handlers
         ├── LogoTextDocumentService  (highlighting, go-to-declaration, rename, code actions)
         └── LogoWorkspaceService    (change signature command execution)
```

Refactorings are modeled as transformations over symbol references rather than text-based edits.

### Key design decisions

#### ANTLR4 parsing.
LOGO does not have a canonical grammar, so Logo.g4 defines a practical subset covering procedures, variables, control flow, turtle commands, and expressions. Keywords are case-insensitive. Statements are structure-based (TO/END, [/]) rather than newline-delimited.

#### Parse-on-change.
The document is reparsed on every change. Given LOGO’s scale, this is fast enough to run synchronously without noticeable latency. Parsed state is stored per file in a ConcurrentHashMap.

#### Symbol table design.
Symbols are tracked per file (LOGO has no module system). Procedures and variables are recorded during a single AST pass. Procedure parameters are scoped to their enclosing definition.

#### Scoped rename.
Parameter rename is limited to the enclosing procedure by restricting traversal to its subtree, avoiding unintended cross-procedure renames.

#### Change signature.
Refactoring is implemented as a transformation over symbol references.
Procedure definitions are rewritten and all call sites updated via a single atomic WorkspaceEdit.
Due to LSP’s lack of structured UI, this is exposed through a predefined “Reverse Parameter Order” code action backed by workspace/executeCommand.

## Change Signature and LSP Limitations

Change signature refactoring requires structured user input (reordering, renaming, adding parameters) and previewing changes across the codebase.

LSP does not provide a standardized mechanism for multi-field interactive UI. Available primitives (`showInputBox`, `showMessageRequest`, `showDocument`) are insufficient for representing a full change signature dialog.

This implementation focuses on the backend transformation:

- rewriting procedure definitions
- updating all call sites
- applying changes atomically via `WorkspaceEdit`

The feature is exposed via `workspace/executeCommand` (`logo.changeSignature`) and demonstrated through a "Reverse Parameter Order" code action.

A production-ready solution would require a client-side extension to provide a structured UI.
